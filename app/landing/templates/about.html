About {{ brand }}

{{ brand }} started as a thin façade over Elasticsearch for heritage institutions. After a full product review (sprints 11 → 27) it grew into a complete publication layer that ingests the file / protocol formats every SIGB, DAMS and archive CMS actually exports, republishes through a clean JSON API and an OAI-PMH endpoint, and lets non-technical operators drive the whole flow from a browser.

Design principles

  • Zero IT required. Every install decision the operator has to make is exposed as a form field in the admin wizard. No YAML editing, no container orchestration, no cron job authoring.
  • Dependencies stay boring. FastAPI + Pydantic + httpx + stdlib XML / CSV. No pymarc, no lxml, no openpyxl, no APScheduler, no message broker — everything bundled in a single Python process.
  • Standards out of the box. OAI-PMH in and out, IIIF manifest passthrough, Dublin Core / LIDO / MARC / EAD mappers, JSON-LD search responses. Aggregators can harvest you the day you install.
  • No scare tactics, no vendor lock-in. No Apple Developer fee, no Authenticode cert, no SaaS control plane. You own the binary, the config, the database and the index.

How it fits together

  • You run {{ brand }} next to your existing Elasticsearch / OpenSearch cluster.
  • You connect your catalogue (SIGB, DAMS, archive CMS) through one of the nine importers.
  • {{ brand }} indexes into ES / OS and exposes a curated public JSON API + an OAI-PMH provider.
  • Aggregators harvest you; your front-end (or a prestataire's front-end) consumes the JSON.
