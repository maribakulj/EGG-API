Who is {{ brand }} for?

It is built for the people who actually curate heritage collections — archivists, librarians, museum registrars — and who do not have a software engineer on call. You run one binary, open the wizard, fill a few fields, and your collection is live on the web with all the right content-negotiation, rate-limiting and audit logging a GLAM institution needs.

Three collection profiles, one public schema

Library — lean Dublin Core shape (id, type, title, description, creators). Fits Koha, PMB, Orphée, Aleph and every OAI-PMH catalogue.
Museum — adds a museum sub-block (inventory number, artist, medium, dimensions, acquisition date, current location) and enables an IIIF manifest passthrough. Fits Axiell, TMS, Micromusée, Mobydoc.
Archive — adds an archive sub-block (unit id, unit level, extent, repository, scope & content, access conditions, parent id). Fits AtoM, Mnesys, Ligeo, ArchivesSpace, PLEADE.

Nine importers, no custom scripts

  • OAI-PMH — Dublin Core: universal SIGB/OAI path.
  • OAI-PMH — LIDO: museums via their DAMS.
  • OAI-PMH — MARCXML: libraries exposing MARC over OAI.
  • OAI-PMH — EAD: archive finding aids over OAI.
  • LIDO flat file: one-shot XML dumps.
  • MARC binary (.mrc, ISO 2709) + MARCXML file with MARC21 or UNIMARC flavors.
  • EAD flat file: EAD 2002 and EAD3.
  • CSV spreadsheet: save as UTF-8 CSV with an id column, EGG ingests.

Every source can be scheduled (hourly / every 6 h / daily / weekly) so the catalogue refreshes itself — no cron job to maintain.

What {{ brand }} is not

  • Not an end-user website or a discovery front-end — it is the JSON API underneath whichever front-end you already use (or the one a prestataire builds on top).
  • Not an indexing engine — it sits in front of your existing search backend (Elasticsearch, OpenSearch) and exposes a curated shape.
  • Not a content management system — your SIGB/DAMS/archives CMS stays the source of truth; {{ brand }} just republishes.
